用户手册(1).docx 与 数据集说明(2).docx 数据修改建议
检查时间：2026-04-25

说明：
- 已抽取两份 Word 的正文和表格内容进行内容级检查。
- 当前环境缺少 soffice / pdftoppm，未做逐页版式渲染，因此本清单主要针对数据、路径、功能口径和文字准确性。
- 两份文档整体方向可用，没有发现大面积推翻重写的问题，主要是若干路径、目录名、流程细节需要更新。

一、用户手册(1).docx 需要修改的点

1. 功能地图里的路由路径要统一加斜杠
- 当前表格中有：report、relationship-space/:pairId、timeline、privacy-security、health-test、attachment-test、challenges、milestones 等。
- 建议统一改为：/report、/relationship-space/:pairId、/timeline、/privacy-security、/health-test、/attachment-test、/challenges、/milestones。
- 原因：当前前端路由文件 web-vue3/src/router/index.js 中实际路径均带 /。

2. 关系空间详情页建议补一个可直接演示的完整 demo 地址
- 当前文档只写了 /demo?route=/pair、/demo?route=/report、/demo?route=/message-simulation。
- 建议新增：/demo?route=/relationship-space/22222222-2222-4222-8222-222222222222。
- 原因：这是当前 README 和演示脚本里更稳定的关系空间详情演示入口，答辩时更容易一步到位。

3. 消息预演演示入口建议补充静态预览入口
- 当前写法：/demo?route=/message-simulation。
- 可保留，但建议同时补充：/preview-demo.html?route=/message-simulation。
- 原因：当前 README 中把该入口作为纯静态跳转演示入口，适合现场快速打开消息预演页。

4. 关系管理里的邀请码建议明确为 10 位
- 当前写法：“系统会生成一组邀请码”。
- 建议改为：“系统会生成 10 位邀请码，用户复制后发给对方。”
- 原因：当前项目设计文档和后端测试口径均为 10 位无易混淆字符的邀请码。

5. 关系维护流程需要补充断绝关系挽留机制
- 当前写法：“发起解绑、取消解绑或确认解绑。”
- 建议改为：
  “发起断绝关系申请，并选择是否给对方挽留机会；若对方选择挽留，可查看对方首条挽留说明与追加留言，再决定接受挽留或仍然断绝；若 24 小时内未回应，或挽留阶段 12 小时内未完成处理，系统会按流程自动解除关系。”
- 原因：当前仓库已实现 pair break request retention flow，这部分比旧的“解绑”更完整。

6. 今日记录部分建议补充“本地预检”
- 当前写法主要描述文字、图片、语音输入。
- 建议补充一句：
  “提交前系统会先在本地识别情绪、风险和隐私线索，图片在高隐私策略下可只保留分析结果。”
- 原因：当前前端 clientAi 已有本地情绪、风险、PII 预检和图片 analysis_only 引导。

7. 关系空间说明建议更新为当前页面重点
- 当前写法：“关系概览或星图、当前信号分与趋势、当前建议动作、主题切换。”
- 建议改为：
  “关系空间详情、关系树能量、今日任务、当前趋势、风险提示、主题切换与下一步行动。”
- 原因：当前实现已经包含关系树能量节点与今日任务，不只是星图概览。

8. 时间轴段落有一个标点错误
- 当前写法：“原文、原图、原语音的保留状态。。”
- 建议删除多余句号，改为：“原文、原图、原语音的保留状态。”

9. 答辩或路演时间口径建议改得更清楚
- 当前写法：“整个演示建议控制在 5-8 分钟内。”
- 如果这是“功能演示”文档，可以保留 5-8 分钟。
- 如果要和 10 分钟答辩 PPT 合并，建议改为：“功能演示环节建议控制在 3-4 分钟，总答辩控制在 10 分钟以内。”
- 原因：你当前准备的是 10 分钟以内答辩材料，避免评委看到两个时间口径。

10. 常见问题中“样例模式不能实时语音”可以保留
- 当前写法准确。
- 建议只补充：“文件语音、实时语音等依赖真实票据或真实接口的能力，以真实账号环境为准。”

11. 使用边界部分可以保留
- “不替代医疗诊断、法律建议、正式心理治疗”的口径是对的。
- 建议继续保留，答辩时这是安全边界加分项。

二、数据集说明(2).docx 需要修改的点

1. 测试临时数据库路径需要修正
- 当前表格写法：backend/test-dbs/、tmp/tests/*.db、tmp/timeline-archive-*/archive.db。
- 建议改为：backend/.test-dbs/、tmp/tests/*.db、tmp/timeline-archive-*/archive.db。
- 原因：当前仓库实际目录是 backend/.test-dbs/，不是 backend/test-dbs/。

2. artifacts 目录描述需要更新
- 当前写法：Artifacts目录下的online-smoke、real-env-smoke等子目录。
- 建议改为：artifacts/online-voice-smoke 保留在线语音冒烟相关结果；真实环境冒烟、手机端双账号、轻量压测等脚本主要保留在 output/。
- 原因：当前 artifacts 目录实际只有 online-voice-smoke；output/ 里才有 frontend_real_env_smoke.py、mobile_account_smoke.py、api_light_pressure.py 等脚本。

3. output 目录说明建议更具体
- 当前写法：“压测、冒烟、PPT重建等输出文件。”
- 建议改为：
  “包含 frontend_real_env_smoke.py、mobile_account_smoke.py、api_light_pressure.py、页面截图、数据库 E-R 图、导出文档等验证与交付产物。”
- 原因：当前 output 目录已经包含 qinjian-database-er.png、qinjian-database-er.mmd、output/doc/ 等交付材料。

4. 本地开发数据库说明建议补充备份文件不要外发
- 当前写法：backend/qinjian.db、backend/manual-auth.db。
- 建议补充：backend/qinjian.db.codex-backup-* 属于本地备份文件，不建议打包提交或外发。
- 原因：当前 backend 目录存在 qinjian.db.codex-backup-20260424-073749。

5. 运行时上传文件位置建议保持双路径
- 当前写法：uploads/、backend/uploads/。
- 这个可以保留。
- 建议补充：正式环境应通过签名访问接口读取，不通过 /uploads/ 公开目录直接访问。

6. 语音回归样本路径可以保留
- 当前写法：tmp/voice-regression/、tmp/voice-sample.wav。
- tmp/voice-regression/ 当前存在，可以保留。
- 如果 tmp/voice-sample.wav 当前不一定存在，建议写成“如 tmp/voice-sample.wav 等临时样本文件”，不要写得太绝对。

7. 时间轴归档样本描述可以保留，但建议强调“临时验证数据”
- 当前写法：tmp目录下的timeline-archive-开头的文件。
- 当前 tmp 下确实有大量 timeline-archive-* 目录。
- 建议补一句：“这些目录通常由归档测试生成，最终交付前可清理，不应作为正式业务数据集。”

8. 五类数据总体分类可以保留
- 前端演示数据、本地开发数据库、自动化测试数据、运行时用户数据、验证截图日志这五类划分是准确的。
- 建议只把目录名和示例更新为当前实际目录。

9. “仓库没有真实关系样本公开语料”的说明必须保留
- 当前写法准确，而且对比赛提交很重要。
- 建议保留原意：当前仓库没有面向外部训练用途的真实关系样本公开语料，演示数据不应被当作训练集外发。

10. 安全与合规段落可以增强一条
- 当前已有：上传目录不公网暴露、签名访问、隐私中心、生产弱配置校验。
- 建议补充：不要把 .env、生产密钥、真实用户上传文件、本地数据库备份文件打包提交。
- 原因：这比单纯说“注意安全”更具体，也符合当前项目目录情况。

三、两份文档共同建议

1. 项目名称统一为“亲健”
- 两份 docx 当前标题均为“亲健”，这个是正确的。
- 后续不要再使用“心健”。

2. 日期建议统一为 2026-04-25
- 如果文档封面或页脚有日期，建议统一成 2026-04-25。
- 原因：README、设计开发文档和当前验证结果都已更新到 2026-04-25。

3. 验证数字如需出现，统一用当前口径
- 后端：python -m pytest --collect-only -q，可收集 223 条测试。
- 前端：npm test -- --run，101/101 通过。
- 构建：npm run build 通过。
- 真实环境冒烟和轻量压测：脚本已保留在 output/，最终答辩前建议重新运行生成当天证据。
- 说明：这两份 docx 当前正文里没有大量旧测试数字，但如果后续补“验证数据”，必须按这个口径写。

4. 外发前建议清理或不打包的内容
