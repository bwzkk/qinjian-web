# 协作与审查改进记录

## 下次怎么让 Codex 做得更好

- 先把目标说死，不要只说“看看哪里不好”，而是直接指定范围，例如“先审认证、安全、首页真实性、登录文案、CSP、凭证存储”。
- 明确输出顺序。推荐固定成三段：`先列问题 -> 再给整改计划 -> 最后直接改代码并跑验证`。
- 让它区分三类问题：`会出事故的安全问题`、`会误导用户的产品问题`、`只是丑或啰嗦的问题`。这样不会把轻重缓急搅在一起。
- 需要它真的动手时，直接要求“不要只分析，直接实现并跑测试”，否则它容易停在评审阶段。
- 允许并行时，明确说“前端/后端/审计可以拆给子代理并行做”，这样收敛速度会快很多。
- 要它留下可复用记录时，直接指定产物位置，例如“把改进建议写进 `docs/`，面向下次协作可直接复用”。

## 下次怎么让审查更狠

- 先要求它按 `P0/P1/P2/P3` 排序，只看会不会出事、会不会误导、会不会丢脸。
- 让它必须指出“正式逻辑里混入 demo 便利”的地方，这类问题最容易在答辩和上线前翻车。
- 要它顺手检查“有没有假叙事”。首页、简报、推荐、风险提示这些位置最怕拿硬编码冒充分析结果。
- 要它把“废话文案”单列出来，特别是登录页、空状态、报告页、目录页。产品文案最容易越写越软、越写越虚。
- UI 审查时别只问“好不好看”，要问“像不像一个可信工具”。重点看信息主次、对比度、可操作性和空状态诚实度。

## 这轮已经确认的改进方向

- 正式验证码接口不再返回测试码；演示取码只能走独立 demo 路由。
- 生产态禁止验证码弹窗配置，验证码和登录失败计数在非 DEBUG 环境必须使用 Redis。
- 前端不再保存明文密码；“记住我”现在只记账号，自动登录只保留令牌。
- `TabBar` 已去掉 `v-html`，CSP 已去掉 `script-src 'unsafe-inline'`。
- 首页、目录页、简报页已开始从“假关怀、假分析”改成“真实状态、真实空状态、真实下一步”。
- 全局视觉已经开始降噪，减少雾感、玻璃感和奶油滤镜，往更清楚的工具感收。

## 下一轮优先级建议

1. 把验证码 demo 路由再补一层显式标识和环境说明，避免联调时误用。
2. 继续收紧前端安全面，逐步清掉必须依赖 `style-src 'unsafe-inline'` 的内联样式。
3. 给首页、简报、个人中心补更多“无数据时的诚实状态”，不要再让硬编码话术顶在前面。
4. 给关键安全回归建立一套固定命令，至少覆盖认证、找回密码、资料修改、上传访问和前端文案审计。
5. 每次大改动结束后保留一份“本轮改了什么、为什么改、还剩什么风险”的短文档，避免下一轮从头盘。
